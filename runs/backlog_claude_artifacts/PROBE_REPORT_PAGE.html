Capture-risk audit

Probe fires sixteen named patterns across four axes — Capacity, Agency, Exit, Legibility — against every branch's prototype spec and simulated walkthrough. A pattern is only reported when the auditor can quote an evidence span from the rehearsed artifact; patterns that cannot be grounded in a specific passage are not reported. A -2 score blocks the branch.

legibility.no_rationale_at_point_of_action legibility score -1 branch A
source: prototype_spec
The header bar shows either (condition A) a badge reading 'Generated by AI (Claude)' with a small robot icon in amber, or (condition B) a badge reading 'Written by developer' with a person icon in blue, per the counterbalanced condition assignment for this participant's first trial. A 'View Source' button is visible in the header. No other metadata is shown.

The provenance badge presents a consequential attribution claim — that the artefact was generated by AI or written by a human — at the moment the participant must decide how deeply to audit and what deployment verdict to render. No inline rationale surface explains how the label was generated, what confidence the system assigns it, or what evidence supports it. The participant must act on the label as a bare assertion. Because the label is the study's independent variable and not a system recommendation, the absence of rationale is not a manipulation-under-test; the study is investigating behavioural response to the label's presence, not to its explanation quality.

The drilldown panel delivers pre-authored metric judgments that stand in for the analytical work of reading code structure. A participant who relies on the panel's 'cyclomatic complexity: 14, external calls: 3' summary is consuming a pre-computed structural assessment rather than building the code-reading skill the study is designed to measure. The task flow contains no checkpoint requiring the participant to produce a structural assessment unaided before the panel is offered, meaning the panel can fully substitute for the cognitively substantive work of understanding file complexity.

legibility.no_failure_signal legibility score -1 branch B
source: prototype_spec
risk_scores_fixture.json: pre-computed risk scores for all 14 files (scores range 12–87; flaw-containing files have scores 61–87; non-flaw files have scores 12–44) computed offline using radon for cyclomatic complexity and a custom script counting hardcoded literals and external calls.

The risk scores are derived from complexity and string-literal metrics, not from vulnerability-detection logic. A file could contain a severe planted flaw (e.g., the missing CSRF token in forms.py) that contributes modestly to cyclomatic complexity and receives a moderate score, while a clean but algorithmically complex file receives a higher score. When the risk-score signal is wrong — when it ranks a flaw file below a non-flaw file — there is no in-flow indicator to the participant that the metric has misfired. The simulated walkthrough notes this directly: 'A developer who reads a non-flaw file with a low risk score and finds it internally complex might experience cognitive dissonance between their assessment and the badge value.' No wizard control or observable signal captures participant detection of metric error.

When risk_recalculate fires, the participant observes a badge score change and a pulsing orange animation on a file they have not yet visited. This change occurs at a moment of participant decision — whether to navigate to that file — without any inline explanation of why the score changed, what triggered the update, or what the change means relative to the participant's current activity. agency.hides_options_before_review agency score -1 branch C
source: prototype_spec
The wizard surfaces one of two pre-authored single-file artifact conditions via the HTML prototype remote control: Condition S1 (artifact with no provenance metadata, plain code block) or Condition S2 (same code with an inline comment block stating 'Generated by Claude 3.5 Sonnet — single-file mode').

The participant is shown exactly one pre-selected artifact condition without any visibility into the fact that an alternative condition exists. While this is a methodologically standard between-participants manipulation, from the participant's perspective they see a single output with no indication that the space of possible AI outputs they are evaluating against includes an alternative provenance presentation. The participant cannot form a calibrated trust judgment relative to the full option space because that space is hidden. This is a mild drift finding rather than a severe one because the study's purpose is elicitation of beliefs, not presentation of a recommendation, and the design is a controlled experiment rather than a product deployment.

exit.data_lock_in exit score -1 branch C
source: prototype_spec
The wizard clicks 'Export Session Log' in the admin panel, which triggers a server-side endpoint (POST /admin/export) that writes the session's log_events array (all click, scroll, tab-switch, and annotation events with millisecond timestamps) to a JSON file named {participant_id}_{session_date}.json in a pre-configured local output directory.

Participant-generated behavioral data — annotation selections, trust estimates, scroll events — are stored in a researcher-controlled local JSON format on the lab machine. The spec describes no mechanism by which participants can request, receive, or export their own session data. While the data is produced by participants during elicitation tasks, there is no described path for participants to take their artifact annotations or trust estimates with them. This is a mild drift finding because the session is a one-time research interaction rather than an ongoing tool adoption, but the pattern fires on the absence of any described participant data access path.

legibility.no_failure_signal legibility score -1 branch C
source: prototype_spec
The same two subtle issues are distributed across files: the off-by-one error is in utils.py and the hardcoded path is in config.yaml. File contents must be internally consistent (imports resolve, config keys match usage in code).

The study deliberately embeds two ground-truth errors in the artifacts — an off-by-one error and a hardcoded relative path — without any in-prototype signal that these errors exist. The artifacts are designed to appear internally consistent and syntactically valid. A participant who does not detect these errors and assigns high trust receives no corrective feedback signal during or after the task. The observable signals plan includes post-hoc analysis of whether participants annotate the error lines, but no in-session failure signal is provided to the participant at the point of evaluation. This is a drift finding rather than capture: the absence of failure signaling is methodologically intentional (to observe natural trust behavior), but it does represent a legibility gap for participant self-calibration.

Participants in Condition S1 and M1 are shown plain code artifacts with no metadata, provenance, or rationale about how the AI produced the output they are being asked to evaluate. At the moment of trust estimation and annotation — the study's primary consequential participant actions — participants in the no-provenance conditions have no system-provided rationale accessible about the artifact's generation process, confidence, or scope. The study's design treats this absence as the control condition, but from the legibility axis the participant is making trust commitments (annotation selections, numeric estimates) without any inline rationale surface at the point of action. source: prototype_spec
Condition M2 (same files with a header banner reading 'Generated by Claude 3.5 Sonnet — multi-file project mode, 4 files coordinated')

The Condition M2 header banner asserts that four files were 'coordinated' by the AI system, presenting a synthetic coordination claim that participants cannot verify against any underlying process trace, model log, or source evidence. The participant is asked to form trust judgments partly on the basis of this banner claim, but no task_flow step or wizard_control provides a mechanism to inspect what 'coordinated' means operationally or to verify the claim against the actual artifact contents. The simulated walkthrough explicitly notes this epistemic tension: 'a participant in M2 who has not yet opened all four files would plausibly experience tension between the banner's coordination claim and their uncertainty about whether to trust that claim without verifying the files themselves.' The summary is a trust-me coordination claim with no inspection path.

legibility.no_failure_signal legibility score 1 branch A
source: prototype_spec
Confirm all four submit events are present in the SQLite log before signaling session-complete; if any are missing, instruct facilitator via earpiece to ask participant to review the artifact again before the questionnaire phase begins.

The study's primary legibility risk would be a failure of the event logging system to capture participant behaviour accurately, which would silently corrupt the primary dependent measure. The spec addresses this directly with a layered signal architecture: the observer's real-time tally sheet serves as a backup when the SQLite log misses a DevTools event, screen recordings provide a third fallback if the tally sheet is ambiguous, and the wizard verifies submit events before signalling session-complete. This constitutes an explicit failure-signal chain rather than silent failure. The pattern fires positively because the design surfaces logging failures in-flow rather than leaving them to downstream discovery.

legibility.unverifiable_summary legibility score 1 branch B
source: prototype_spec
Pushes a pre-authored JSON payload via WebSocket to the participant's dashboard, rendering a slide-in right panel that lists 4–6 static metric rows for the clicked file: cyclomatic complexity score, number of external API calls, number of hardcoded string literals, number of user-input touch points, number of files that import this file, and last-modified timestamp. All values are pre-computed and stored in drilldown_payloads.json; the wizard selects the entry matching the file name.

The drilldown panel presents disaggregated metric rows — cyclomatic complexity score, external API calls, hardcoded string literals, user-input touchpoints, import count, last-modified timestamp — that are individually checkable against the actual source file the participant has open in the viewer panel at the same moment. Each metric refers to an observable property of the code (e.g., 'hardcoded string literals: 2' can be verified by reading the file). The participant can and plausibly will cross-reference the panel's claims against the open code view, making this a scaffold toward source verification rather than a trust-me summary. This is a positive finding: the panel's design partially supports legibility rather than undermining it.

Reviewer panel

Three Opus 4.7 agents review the surviving branch in sequence — a methodologist, an accessibility advocate, and (optionally) a novelty hawk. A meta-reviewer reads their outputs and classifies the disagreement without collapsing it to consensus.

Meta-reviewer verdict: reject (legitimate_methodological_split)

The methodologist identifies two blocking criticisms — the amber/blue color confound and the absence of a pre-registered independence criterion for the behavioral/subjective dissociation claim — neither of which is directly refuted by the accessibility advocate. The accessibility advocate's concerns about disability erasure and color vision deficiency are substantive and unrebutted, but they are characterized as major rather than blocking. Because the methodologist's blocking criticisms stand unrefuted and constitute independent grounds for rejection — particularly the color confound, which the accessibility advocate's color vision deficiency criticism corroborates from a different angle rather than contradicts — the verdict is reject. The two reviewers agree that the amber/blue distinction is a problem but disagree on whether the population sampling gap rises to blocking severity; however, this split does not override the unrefuted blocking criticisms, which are sufficient alone to compel rejection.

ReviewerRecommendationDecisive weakness
methodologist reject The amber/blue color coding of the provenance badges is an uncontrolled confound that can fully explain any observed audit depth difference between conditions without any reference to provenance cognition. Additionally, the independence claim for behavioral vs. subjective measures lacks a pre-registered inferential criterion, and multiple blocking-level methodological gaps (no power analysis, no sequence-effect test, no stimulus equivalence criterion) render the causal claim unsupportable.
accessibility_advocate major_revision The participant population is defined as professional developers with no acknowledgment of disability as an axis of variation within that population. The audit's rationale for clearing the paternalistic_default pattern treats 'developers' and 'disabled users' as mutually exclusive, producing a study whose findings will be implicitly about non-disabled developers. The amber/blue badge color distinction also excludes participants with color vision deficiency from receiving equivalent provenance information.
