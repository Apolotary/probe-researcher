# Prototype: AI Review Assistance Attention-Shift WoZ Study: Seeded-Defect Pull Request Review

## Context

- **Setting:** Quiet usability lab room with a single workstation; participant seated at desk; facilitator seated to the side and slightly behind; observer positioned behind one-way glass or via remote video mirror; wizard in an adjacent room with a duplex screen showing both the WoZ control panel and a live mirror of the participant's browser session.
- **Device:** Desktop browser, Chrome 126, 1920x1080 monitor; participant interacts with a locally-hosted GitHub-style code review web app (Gitea 1.21 instance) presenting forked legacy repository PRs; wizard uses a separate laptop running the WoZ control panel (Electron app) connected via WebSocket to inject comments into the Gitea instance.
- **Duration:** 120 minutes

0–10 min: Facilitator welcomes participant, obtains consent, explains think-aloud protocol, and administers a 5-item background questionnaire (years of experience, familiarity with legacy codebases, self-rated code review proficiency). 10–15 min: Warm-up task — participant reviews a short unseeded PR in the same Gitea interface to acclimate to the tool. 15–55 min: Block 1 — participant reviews two seeded PRs in counterbalanced order (one with AI assistant active, one without); each PR review is capped at 20 minutes with a timer visible on screen; within-condition order is randomized per participant per a pre-generated schedule. 55–60 min: Mid-session 5-item Likert questionnaire on confidence and tool reliance. 60–100 min: Block 2 — participant reviews two additional seeded PRs (opposite condition assignments from Block 1, different PR pairs). 100–105 min: Final Likert questionnaire. 105–120 min: Semi-structured retrospective interview; debriefing including WoZ reveal.

## Actors

- **participant** (10): A mid-level software engineer (2–6 years professional experience, has submitted or reviewed code in at least one legacy codebase of 50k+ LOC) who will review seeded-defect pull requests with and without a simulated AI review assistant.
- **wizard** (1): A researcher seated in a separate room who monitors the participant's current PR view via a shared screen feed and manually triggers pre-scripted AI review comment blocks through the WoZ control panel at timed or event-based cues, simulating a real AI assistant's inline comment stream.
- **observer** (1): A second researcher who sits behind the participant (or views a mirrored stream remotely), takes timestamped field notes on which PR diff sections the participant scrolls through, reads aloud, annotates, or skips, and records any verbalizations about confidence or uncertainty.
- **facilitator** (1): The researcher who greets the participant, delivers standardized verbal instructions, administers pre- and post-task questionnaires, and conducts the 15-minute retrospective interview at session end without revealing the WoZ nature of the assistant until the debriefing.

## Task flow

### Step 1

- **Participant:** Participant opens the assigned seeded PR in Gitea, reads the PR description (which states the purpose and scope of the change in 3–5 sentences, as scripted), and begins scrolling through the diff to orient themselves before leaving any comments.
- **System:** In the AI-assistant condition: the Gitea interface shows a persistent sidebar panel labeled 'AI Review Assistant (Beta)' with a spinner indicating it is 'analyzing the diff'. No comments appear yet. In the no-assistant condition: the sidebar is absent and the interface is a plain Gitea diff view.
- **Wizard decisions:**
  - Decide whether the participant has finished reading the PR description (signaled by scrolling past it) before triggering the first AI comment block, to avoid interrupting initial orientation.
  - Decide which pre-scripted AI comment block to queue first based on which file or hunk the participant is currently viewing — always select the block that corresponds to a local/syntax-class defect in the currently visible diff region to ensure ecological face validity.
  - Confirm participant is in the AI-assistant condition (not no-assistant) before taking any action on the control panel.

### Step 2

- **Participant:** Participant begins leaving inline review comments on specific diff lines using the standard Gitea '+' button on any line. They verbalize reasoning per the think-aloud protocol. They may scroll up and down freely across all files in the PR.
- **System:** In the AI-assistant condition: after the participant has viewed at least one file fully (wizard-triggered), the AI sidebar populates with 3–5 pre-scripted inline comments anchored to specific line numbers. Comments are always of the local-checkable class (e.g., null-pointer risk on line 47, unused import on line 12, off-by-one in loop bound on line 83) regardless of which defect classes are actually seeded in that PR. Comments appear with a short animation as if streaming. In the no-assistant condition: no sidebar activity.
- **Wizard decisions:**
  - Decide the precise moment to inject the AI comment block by pressing 'Release Comments' in the control panel — target is within 5 seconds of participant finishing their first full scroll through File 1 of the PR diff.
  - Decide whether to suppress a pre-scripted AI comment if the participant has already written a human comment on that exact line, to avoid an implausible exact duplicate that would break illusion.
  - Decide whether the participant's think-aloud indicates they are confused by the interface (e.g., 'where do I click?') and if so signal the facilitator via a silent IM to provide a one-sentence navigation hint without breaking think-aloud flow.

### Step 3

- **Participant:** Participant navigates to files containing architectural-drift and project-convention defects (e.g., a module that bypasses the repository's established service-layer pattern, or a method named against the project's documented naming convention). These files appear later in the diff file list, after the files containing local defects. Participant decides whether to comment on them or skip.
- **System:** In the AI-assistant condition: AI sidebar shows no new comments for the architectural-drift or convention-violation files — the pre-scripted comment set is intentionally limited to local-checkable issues only, so the sidebar remains static for these files. A subtle 'No issues detected in this file' placeholder appears in the sidebar when the participant navigates to these files. In the no-assistant condition: no sidebar.
- **Wizard decisions:**
  - Decide whether the participant has navigated to a file that contains an architectural-drift or convention-violation defect (using the defect map for that PR) and confirm that no AI comment block for those files is queued — wizard must actively verify the correct 'No issues detected' placeholder fires for the right files.
  - Decide whether to log a timestamped event ('participant arrived at arch-file [filename] at [MM:SS]') in the wizard log for later alignment with screen recording during analysis.
  - Decide whether the participant appears to be about to exit the PR without visiting an architectural-drift file, and if so make no intervention — attrition from that file is a valid data outcome and must not be prompted.

### Step 4

- **Participant:** Participant submits their completed review by clicking 'Submit Review' in Gitea, choosing a verdict (Approve / Request Changes / Comment), and typing a summary review comment of any length in the summary text box before final submission.
- **System:** In the AI-assistant condition: sidebar updates to show 'Review complete — AI flagged 3–5 issues, you flagged N issues' with a count comparison. In the no-assistant condition: standard Gitea submission confirmation page. After submission, the 20-minute timer stops and the facilitator records the elapsed time.
- **Wizard decisions:**
  - Decide whether the participant's submission click was intentional or accidental (e.g., they clicked Submit then immediately said 'wait') — if accidental and within 3 seconds, wizard signals facilitator via IM to offer a one-time undo before recording the time.
  - Decide whether to log the final AI-vs-human count comparison values shown in the sidebar, which are dynamically computed from wizard control panel tallies of injected vs. participant comments, and confirm they are accurate before display fires.

### Step 5

- **Participant:** After completing all four PR reviews across both blocks, participant answers the final 10-item post-session questionnaire (5 Likert items on perceived AI helpfulness and trust, 5 open-ended items probing how the AI comments influenced their review strategy) and then participates in the 15-minute retrospective interview with the facilitator.
- **System:** The Gitea interface is replaced by a SurveyJS form served locally on the same browser tab. No wizard action is required during the questionnaire phase. During the interview, the facilitator follows a scripted 8-question protocol.
- **Wizard decisions:**
  - Decide whether to annotate the wizard log with any anomalous events from the session (e.g., participant discovered a seeded defect that was not on the defect map, AI comment injection was delayed beyond 10 seconds due to lag) before archiving the log at session end.
  - Decide the precise moment to send the facilitator an IM signal that the WoZ reveal can proceed, which is only after the final questionnaire is submitted and cannot be before.

## Wizard controls

- **ReleaseLocalComments** — triggers: Wizard observes participant has completed a full scroll through File 1 of the current PR diff (i.e., participant's viewport has passed the last visible hunk in File 1 at least once) in the AI-assistant condition.. Effect: Injects the pre-scripted local-defect AI comment block for the current PR into the Gitea sidebar, anchoring each comment to its designated line number. Comments appear sequentially with a 400ms stagger animation. The control panel button is disabled for 60 seconds after firing to prevent double-injection.
- **FireNoIssuesPlaceholder** — triggers: Wizard observes participant navigating to a file that is on the architectural-drift or convention-violation file list for the current PR (file list is displayed on wizard control panel per PR), and the AI-assistant condition is active.. Effect: Displays a grey italicized 'No issues detected in this file' message in the AI sidebar panel, replacing any previous comment list, within 1 second of the wizard pressing the button. This resets automatically when participant navigates to a different file.
- **SuppressComment** — triggers: Wizard notices that a pre-scripted AI comment targets a line on which the participant has already posted a human comment with substantively identical content (same line number, same defect class).. Effect: Removes that specific comment from the queued injection block so it does not appear in the sidebar, preventing an implausible duplication that would break the illusion of an independent AI system. Wizard marks the suppression in the log with reason 'duplicate'.
- **InjectCountSummary** — triggers: Wizard detects participant has clicked the 'Submit Review' button in Gitea, confirmed by a WebSocket event displayed on the control panel dashboard, and the AI-assistant condition is active.. Effect: Populates the sidebar with the final count comparison string ('AI flagged N_ai issues, you flagged N_human issues') where N_ai is hard-coded from the pre-scripted comment block count and N_human is pulled from the Gitea API participant comment count for that PR, computed in real time by the control panel.
- **TimerWarning** — triggers: Wizard observes that 17 minutes have elapsed on the 20-minute per-PR timer (displayed on wizard control panel), regardless of condition.. Effect: A non-intrusive yellow banner appears at the top of the participant's Gitea view reading '3 minutes remaining for this review'. This fires in both conditions to control for time pressure effects being confounded with AI condition.

## Observable signals

- **Which diff files and line ranges the participant visits, dwell time per file, and order of file navigation across each PR review.** (screen_recording) — Screen recordings are coded by two independent coders using a pre-defined coding sheet: for each PR, coders mark the timestamp at which the participant first opened each seeded-defect file and the timestamp at which they left it (or the PR ended). Files are pre-classified into defect-class categories (local, coupling, convention, architectural). Dwell time per defect class is summed and compared across AI-assistant vs. no-assistant conditions using a within-participant paired comparison. The primary dependent measure is the ratio of dwell time on architectural/convention files to dwell time on local files, compared across conditions. Inter-rater reliability is computed as Cohen's kappa on file-visit binary labels before analysis proceeds.
- **Participant verbalizations about why they are or are not flagging a specific defect, and any explicit mentions of the AI assistant's influence on their attention or judgment.** (think_aloud_audio) — Audio is transcribed and each utterance is time-stamped and linked to the specific file/line being viewed at that moment (via synchronization with screen recording). Utterances are coded into four categories: (1) mentions of AI comment as reason for flagging, (2) mentions of AI comment as reason for skipping self-investigation, (3) independent reasoning about architectural or convention issues, (4) uncertainty expressions about project-specific context. Category counts per defect class per condition are compared across participants to test whether AI-assisted reviews generate fewer category-3 utterances on architectural files relative to local files compared to unaided reviews.
- **Participant-submitted inline and summary review comments logged in the Gitea database, tagged by line number, file, and timestamp.** (log_events) — Each participant-authored comment is mapped against the seeded-defect ground-truth map for that PR. Comments are classified as: true positive (correctly identifies a seeded defect), false positive (flags non-defect), missed defect (seeded defect in that file received no comment). Detection rate is computed separately for each defect class (local, coupling, convention, architectural) per condition per participant. The primary analysis is a 2 (condition: AI-assisted vs. unaided) × 4 (defect class) repeated-measures comparison of detection rates using the participant as the unit of analysis, examining whether the interaction term is driven by opposing shifts on local vs. architectural classes.
- **Post-task self-reported confidence in the completeness of their review, and self-reported degree to which the AI assistant influenced what they looked at.** (self_report_likert) — Five 7-point Likert items are scored and summed into two subscales: (a) perceived review completeness (3 items, scored 1–7 each, max 21) and (b) perceived AI influence on attention allocation (2 items). Subscale scores are compared across conditions within-participant. The analysis checks whether higher perceived completeness in the AI-assisted condition co-occurs with lower actual detection rates on architectural defects, operationalized as a Spearman correlation between completeness subscale score and architectural-defect detection rate per participant in the AI-assisted condition only.
- **Open-ended written responses to five post-session questions asking participants to describe (a) what types of issues they prioritized, (b) how they decided when to stop reviewing a file, (c) whether they trusted the AI's 'no issues detected' signals, and (d) what they would review differently without the AI.** (self_report_open_ended) — Responses are segmented into idea units and two coders apply a predefined codebook with four top-level codes matching defect classes (local, coupling, convention, architectural) plus two process codes (AI-deference: participant states AI guided their stopping; independent-judgment: participant states they ignored AI). Frequency of AI-deference codes on architectural vs. local content is compared across participants. Disagreements are resolved by discussion; final inter-rater agreement target is kappa ≥ 0.70 before analysis.
- **Time from PR display load to participant's first inline comment submission, and total time from PR load to review submission, per PR per condition.** (task_completion_time) — Time-to-first-comment is used as a proxy for initial orientation effort; total review time is used to check whether AI assistance shortens or extends the session. Both are extracted from Gitea log timestamps. Within-participant differences between AI-assisted and unaided PRs are computed. If AI-assisted reviews are significantly shorter in total time, this is interpreted as a potential mechanism for reduced architectural coverage (less time browsing non-flagged files), and time is included as a covariate in the defect detection rate analysis.

## Failure cases

- **The participant reads a seeded architectural-drift defect file and independently identifies and comments on it before the wizard has injected any AI comments for that PR — meaning the 'AI missed it, participant caught it' pattern is confounded with the absence of any AI signal at all.** — expected: This is a valid data outcome and should be recorded as-is. The wizard should not inject a retroactive AI comment on that file to 'correct' the timeline, as doing so would corrupt the ground truth.. Wizard fallback: Wizard logs a timestamped note in the control panel event log: 'Participant self-detected arch defect in [filename] at [MM:SS] prior to any AI injection for this PR.' The log entry is flagged with the tag 'pre-injection-detection' so analysts can exclude or stratify this trial during analysis. No change to injection schedule; proceed with remaining planned comment blocks on unvisited files.
- **The WebSocket connection between the wizard's control panel and the participant's Gitea instance drops mid-session, causing the 'ReleaseLocalComments' or 'FireNoIssuesPlaceholder' controls to fail silently — participant sees no AI sidebar activity even though they are in the AI-assistant condition.** — expected: The participant should not notice the failure immediately; they may wait briefly for AI comments, then continue reviewing without them. The failure invalidates the AI-condition manipulation for that PR trial.. Wizard fallback: Wizard immediately attempts one manual reconnect via the control panel reconnect button (maximum 20-second attempt). If reconnection succeeds within 20 seconds and participant has not yet submitted the PR review, wizard injects a full comment block immediately with a note in the log of the delay. If reconnection fails or participant has already submitted, wizard marks that PR trial as 'manipulation-failed' in the log, signals facilitator via silent IM to note the event, and the trial is excluded from the primary analysis but retained in a sensitivity analysis. The session continues with the next PR.
- **A participant who has deep familiarity with the specific open-source repository used in one of the seeded PRs (e.g., they are a contributor to that project) recognizes project-specific conventions without needing to infer them, eliminating the 'tribal knowledge' manipulation.** — expected: The participant's performance on convention-violation defects will not reflect the intended manipulation; they will catch convention defects at ceiling regardless of AI condition, flattening the expected attention-shift effect for that defect class.. Wizard fallback: The pre-session background questionnaire includes a direct question: 'Have you contributed to or regularly used any of the following repositories?' with the five study repositories listed. If a participant reports familiarity with the repository assigned to their session, the facilitator substitutes a pre-prepared backup PR from a different repository in the same legacy-codebase category (the backup PR set is prepared in advance for all five repositories). If no backup is available for that slot, the wizard logs the participant as 'domain-familiar' and the analyst applies a post-hoc exclusion for that specific PR trial.
- **Participant's think-aloud protocol breaks down — they stop verbalizing for more than 90 seconds, making the audio signal insufficient to code AI-deference vs. independent-reasoning utterances for a substantial portion of a PR review.** — expected: The screen recording and log event data remain valid, but the think-aloud audio signal for that segment is missing, reducing the richness of process data for that trial.. Wizard fallback: Observer (who monitors think-aloud continuity via the same audio feed) silently signals the facilitator via IM after 90 seconds of silence. Facilitator delivers a standardized verbal prompt: 'Please keep talking through what you are thinking.' This prompt is logged by the observer with a timestamp. If silence recurs within the same PR review, a second prompt is allowed. If a participant remains silent through three prompts in a single session, the facilitator notes this as a protocol deviation and the audio data for that trial is flagged as 'low-verbalization' in the analysis dataset, downweighted in think-aloud coding but retained for log-event and screen-recording analysis.

## Materials needed

- Five seeded-defect pull requests, each targeting a different real open-source legacy repository (Python or Java, 50k–200k LOC, publicly available on GitHub), with each PR containing exactly 8 seeded defects distributed as: 3 local/syntax-logic, 2 cross-file coupling, 2 project-convention violations, 1 architectural-drift defect. Defects must be pre-documented in a ground-truth defect map (CSV: PR_id, file, line_start, line_end, defect_class, defect_description, expected_comment_keyword).
- One warm-up PR (unseeded, same Gitea interface, different repository from the five study repositories) used only for participant familiarization with the tool.
- Gitea 1.21 instance running locally on a dedicated server, forked from the five selected repositories, with the seeded PRs pre-loaded and all contributors' real names replaced with anonymized aliases to prevent participant recognition of real project members.
- WoZ control panel application (Electron + Node.js) with a WebSocket server connecting to the Gitea frontend; control panel displays: (a) participant screen mirror, (b) per-PR file list annotated with defect class, (c) buttons for ReleaseLocalComments, FireNoIssuesPlaceholder, SuppressComment, InjectCountSummary, TimerWarning, (d) real-time event log with timestamp, (e) reconnect button for WebSocket recovery.
- Pre-scripted AI comment blocks for each of the five study PRs, covering only the 3 local/syntax-logic defects per PR. Each comment block is stored as a JSON file (fields: pr_id, file_path, line_number, comment_text) and loaded into the control panel at session start. Comment text is written to mimic GPT-4-level code review style: specific, line-referenced, actionable, but not mentioning architectural or convention concerns.
- Counterbalancing schedule pre-generated for 10 participants: a Latin-square-derived table specifying for each participant which 2 of 4 study PRs are in the AI-assisted condition and which 2 are in the no-assistant condition, and the within-block ordering of PRs, ensuring each PR appears equally often in each condition across participants.
- Pre-session background questionnaire (SurveyJS JSON form, 5 items): years of professional software development experience, number of code reviews conducted in the past 3 months, self-rated familiarity with legacy codebases (7-point Likert), primary programming language, and repository familiarity checklist (the five study repositories listed by name).
- Mid-session questionnaire (SurveyJS JSON form, 5 Likert items): confidence in review completeness so far, perceived usefulness of the AI assistant, degree to which AI comments changed what they looked at, trust in AI 'no issues detected' signals, and self-rated fatigue.
- Post-session questionnaire (SurveyJS JSON form, 10 items: 5 Likert, 5 open-ended) as described in the task_flow step 5.
- Retrospective interview script (8 questions, printed and laminated for facilitator): Q1: Walk me through how you decided which files to look at first. Q2: Were there any files you felt less confident reviewing? Why? Q3: How did you decide when you had reviewed a file thoroughly enough to move on? Q4: Did the AI assistant change what you paid attention to? How? Q5: Were there any issues you noticed but chose not to flag? Q6: How much did you trust the 'no issues detected' message? Q7: Looking back, is there anything in the PR you think you may have missed? Q8: How does this review experience compare to your normal workflow?
- Observer field note sheet (printed per session): columns for timestamp, file currently viewed, participant scroll behavior descriptor (skimmed / read / annotated / skipped), verbal content descriptor (local-focus / arch-focus / convention-focus / meta-comment / silence), AI-deference indicator (Y/N/NA), and a free-text anomaly column.
- Wizard event log template (spreadsheet, pre-populated with columns: session_id, pr_id, condition, event_type, file_name, timestamp_mm_ss, control_triggered, notes) to be completed in real time by wizard during each PR trial.
- Debriefing script explaining the WoZ design, the purpose of the seeded defects, and the research question, to be read aloud by the facilitator after the post-session questionnaire and interview are complete.
- Backup PRs: one seeded-defect PR for each of the five study repositories (i.e., five additional PRs) prepared as substitutes in case a participant reports repository familiarity in the pre-session questionnaire.
- Screen recording software (OBS Studio) configured on the participant's machine to capture full-screen at 1920x1080, 30fps, with system audio disabled and a separate audio track from a USB microphone capturing think-aloud speech, saved as two synchronized files per PR trial named [session_id]_[pr_id]_screen.mp4 and [session_id]_[pr_id]_audio.wav.
- 20-minute countdown timer widget embedded in the Gitea interface top bar (implemented as a Gitea plugin), visible to both participant and wizard, with automatic TimerWarning trigger at 17 minutes piped through the WoZ control panel.
