Capture-risk audit

Probe fires sixteen named patterns across four axes — Capacity, Agency, Exit, Legibility — against every branch's prototype spec and simulated walkthrough. A pattern is only reported when the auditor can quote an evidence span from the rehearsed artifact; patterns that cannot be grounded in a specific passage are not reported. A -2 score blocks the branch.

capacity.substitutes_for_practice capacity score -1 branch B
source: prototype_spec
Step 3: participant_action: 'The participant listens to the wizard's response as it is read by the screen reader from the chat log. The participant then either asks a follow-up query (returning to step 2), or navigates to the answer-submission form and types their outlier identification answer…'

The conversational agent performs the core data-interpretation work — identifying which data point is an outlier, reading exact numeric values, characterizing trends — on behalf of the participant at every turn. The participant's cognitive contribution is reduced to query formulation and response acceptance; the skilled action under study (interpreting chart data from a screen-reader-accessible representation) is never performed unaided during the experimental blocks. There is no task-flow step requiring the participant to synthesize the underlying data without wizard assistance before arriving at their final answer.

capacity.no_fade_mechanism capacity score -1 branch B
source: prototype_spec
wizard_controls: ['inject_response', 'inject_clarifying_question', 'one_minute_warning', 'inject_error_recovery'] … The static description div is rendered as a standard ARIA live region … No wizard action is required at this step; the static description is pre-authored for each of the three chart stimuli and served directly from the HTML fixture file.

All four wizard controls are available at full strength throughout the entire session. There is no described mechanism — conditional gating, confidence threshold, or explicit 'training wheels off' mode — that would reduce wizard assistance as a participant demonstrates growing familiarity with the interface across the three blocks. The interblock review period (step 4 wizard decisions) explicitly prepares the wizard for the next block at the same level of support, not at a reduced level.

agency.hides_options_before_review agency score -1 branch B
source: prototype_spec
Step 3 system_response: 'Each wizard-composed response is injected into the chat log div via a JavaScript function that appends a new list item with role='listitem' inside the ARIA live region, triggering an automatic screen-reader announcement.' … wizard_decisions step 2: 'Decide whether the query is ambiguous … and, if so, whether to ask a single clarifying question or answer the most probable interpretation directly.'

When the wizard selects the 'most probable interpretation' of an ambiguous query, the participant receives a single answer grounded in one interpretation without being shown that the wizard considered and rejected alternative interpretations. The participant cannot inspect the alternative readings the wizard chose among before acting on the single response. No wizard control surfaces the set of candidate interpretations to the participant at the point of decision.

exit.data_lock_in exit score -1 branch B
source: prototype_spec
materials_needed: 'Wizard console application: a localhost web app (Node.js + Express) running on the wizard's machine that … writes all events to a local SQLite file named session_{participant_id}_{date}.db.' … 'Screen-recording software (OBS Studio, configured to capture both the participant's display and the microphone audio channel) running on the participant's machine, with output saved as participant_{id}_block_{n}.mkv.'

Session artifacts — conversation logs in SQLite and screen recordings in MKV — are stored in researcher-controlled local files. No export mechanism, data-return pathway, or description of participant access to their own session data is present anywhere in the spec. A participant who wished to retain their interaction history (e.g., for their own review of how they queried the chart) has no described path to do so. The data lock-in here affects participant access to their own produced content rather than vendor lock-in, but the pattern still applies.

legibility.no_failure_signal legibility score -1 branch B
source: prototype_spec
failure_cases scenario 3: 'The participant's submitted answer may omit or misstate the numeric value without any verbal signal of uncertainty, making the error invisible in the think-aloud stream and leading to an incorrect accuracy score that looks like a system failure rather than an audio presentation failure.' … step 3 wizard_decisions: 'Decide whether to volunteer an unsolicited clarification if the participant appears to be about to submit a clearly wrong answer due to a wizard response error (policy: do NOT volunteer; log the error for debrief).'

The spec explicitly identifies the scenario where a participant mishears a numeric value and proceeds to submit a wrong answer 'without any verbal signal of uncertainty' — meaning the error is invisible to the participant in the flow. The wizard non-intervention policy preserves ecological validity but confirms there is no in-flow failure signal to the participant when a response has been misheard or misunderstood. The observable_signals do not include any measure of participant detection of wizard response errors during the task.

legibility.no_rationale_at_point_of_action legibility score -1 branch B
source: prototype_spec
step 3 participant_action: 'The participant listens to the wizard's response as it is read by the screen reader from the chat log. The participant then either asks a follow-up query (returning to step 2), or navigates to the answer-submission form and types their outlier identification answer…' … wizard_controls inject_response effect: 'Calls a JavaScript function on the participant's browser … that removes the 'Thinking…' placeholder element, appends a new list item containing the wizard's response text to the ARIA live region chat log.'

At the moment the participant decides whether to accept a wizard response and commit to a final answer, no inline mechanism exposes how the wizard arrived at the stated value — whether it was read from a data table, estimated, or inferred from a trend. The response text is injected as a plain string with no provenance indicator, confidence marker, or 'why this answer' affordance accessible before the participant presses submit. The participant acts on 'trust me' output at the consequential decision point.

legibility.unverifiable_summary legibility score -1 branch B
source: prototype_spec
step 2 system_response: 'The interface appends the participant's query to a visible chat transcript region (ARIA live region, role='log', aria-live='polite') and immediately displays a 'Thinking…' status message that the screen reader announces.' … step 3 system_response: 'Each wizard-composed response is injected into the chat log div via a JavaScript function that appends a new list item with role='listitem' inside the ARIA live region, triggering an automatic screen-reader announcement.'

The wizard's responses synthesize information from a ground-truth data table that is physically present only in the wizard room and is never surfaced to the participant in any form. The participant receives a claim such as 'Product Line C drops to 120K in March' with no mechanism to verify that value against the underlying chart data — because the chart is a visual SVG/Canvas element with no alt text, accessible only through the static ARIA description and subsequent conversational turns. The conversational interface is the sole access path, making every wizard summary inherently unverifiable by the participant at the point of action.

capacity.no_rehearsal_space capacity score 1 branch B
source: prototype_spec
Minutes 10–20: Facilitator introduces the think-aloud protocol and runs a warm-up task (querying a conversational interface about a simple bar chart that is NOT one of the study charts) so the participant learns the interaction mechanics and the wizard calibrates response latency.

The session structure explicitly includes a ten-minute warm-up phase using a non-study chart whose purpose is for the participant to learn the interaction mechanics before any experimental block begins. This constitutes a low-stakes rehearsal space where errors carry no consequence for study data. The warm-up is isolated from scored task blocks, providing genuine practice opportunity before consequential performance is measured.

agency.paternalistic_default agency score 1 branch B
source: prototype_spec
actors: 'A blind or low-vision adult who uses a screen reader (NVDA or JAWS on Windows, or VoiceOver on macOS) as their primary means of computer access' … context setting: 'No sighted assistance is available to the participant during tasks.' … materials_needed: 'device: Windows 10 laptop, Chrome 124 … or macOS 13 laptop, Safari 17 … matched to participant's home screen reader.'

The study population is blind and low-vision adults — a group historically subject to paternalism in assistive technology design. Critically, the spec explicitly counteracts paternalistic default by matching the lab environment to each participant's own screen reader and configuration, requiring at least one year of daily screen-reader use as an inclusion criterion, and withholding sighted assistance during tasks. These are affirmative design choices that treat participants as competent expert users of their own tools rather than defaulting to protective interventions. The Kafer pattern fires positively here as a scaffold.
