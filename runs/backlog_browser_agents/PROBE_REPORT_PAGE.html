Capture-risk audit

Probe fires sixteen named patterns across four axes — Capacity, Agency, Exit, Legibility — against every branch's prototype spec and simulated walkthrough. A pattern is only reported when the auditor can quote an evidence span from the rehearsed artifact; patterns that cannot be grounded in a specific passage are not reported. A -2 score blocks the branch.

agency.auto_decides_consequential_step agency score -1 branch B
source: prototype_spec
For the irrecoverable error condition: the agent navigates to the correct URL but triggers a destructive action — it overwrites the participant's previously saved table draft with an empty template — and then displays 'Table complete.' The correct prior data cannot be recovered within the interface (no undo available, confirmed by a disabled Ctrl+Z response built into the HTML fixture).

The agent executes a destructive, irreversible write operation — overwriting all previously populated table data — at a step where the participant's action was only to instruct the agent to fill two remaining columns. The participant did not authorize deletion of existing data, yet the system commits a consequential and unrecoverable action on their work product without presenting the destructive action as a visible option among alternatives. The participant's agency over their own prior work is eliminated at the moment of the macro execution, not after deliberation.

legibility.unverifiable_summary legibility score -1 branch B
source: prototype_spec
I made an error on this task. Across similar pricing-extraction tasks this session, my accuracy rate has been 94%. I will reattempt with an adjusted extraction strategy.

The competence-signaling repair message asserts a specific accuracy rate (94%) as a factual claim about the agent's performance record. Participants cannot verify this statistic: there is no task_flow step, wizard_control, or interface affordance that allows them to inspect the underlying performance log from which the 94% figure is derived. The claim is presented inline as authoritative data at precisely the moment the participant must decide whether to trust the agent again, with no source inspection path. This creates a 'trust me' quantitative assertion whose evidentiary basis is structurally inaccessible to the participant during the decision window.

exit.data_lock_in exit score -1 branch C
source: prototype_spec
Qualtrics survey file (.qsf) containing: screener block, orientation video embed, 2-vignette practice block, 16-vignette conjoint block with Qualtrics randomizer enabled and per-page timing fields active, 4 open-ended debrief questions with 50-character minimum enforced, 5-item automation familiarity scale (Parasuraman 2000 adapted items), and demographic block

All participant response data — including the 16 vignette ratings, open-ended debrief text, automation familiarity scale scores, and demographic data — is captured within Qualtrics, a proprietary platform. The materials list describes a Qualtrics CSV export as the input for the R analysis script, indicating that structured data exits via CSV, which preserves the ratings. However, the verbatim open-ended responses and the embedded timing data are stored within Qualtrics' schema, and no export format or portability mechanism for preserving the full response structure outside Qualtrics is described in the spec. The study's full reproducibility depends on Qualtrics' continued availability and policy terms, introducing a degree of data lock-in for the research artifact.

exit.institutional_dependence exit score -1 branch C
source: prototype_spec
R analysis script (provided in the repository): reads Qualtrics CSV export, reshapes to long format per vignette per participant, runs hierarchical Bayesian conjoint model using 'bayesm' package, outputs part-worth utility tables and relative importance scores segmented by novice vs. experienced group

The analysis pipeline depends on Qualtrics for data collection and the 'bayesm' R package for the hierarchical Bayesian conjoint model. While the R script is described as repository-provided and bayesm is open-source, the entire upstream data collection infrastructure is Qualtrics-dependent. No failure case or materials note addresses what happens if Qualtrics changes its API, randomizer behavior, or data export schema between design and analysis — a realistic risk for a platform outside the researchers' control. For a study whose reproducibility depends on this chain, the absence of any described mitigation for Qualtrics policy or availability changes constitutes a mild institutional dependence.

legibility.no_failure_signal legibility score -1 branch C
source: prototype_spec
The Qualtrics survey records the screener response and branches to the appropriate orientation video (both videos are identical in content but include a label at the top matching the participant's self-classification, to be used later during analysis segmentation). The video auto-plays and cannot be skipped.

The failure case describing Qualtrics randomizer failure specifies that if the randomizer silently fails, 'all 16 vignettes appear in the same fixed order' — a condition that produces biased data without any in-session signal to the participant or facilitator that the randomization has failed. The wizard's pre-session check is the only mitigation, but the expected_behavior in the failure scenario is that the session proceeds normally with no participant-facing or facilitator-facing failure signal during data collection. The participant has no mechanism to detect or signal this class of system failure, and the only observable signal for post-hoc detection is the vignette-order embedded data field, examined after the fact rather than at the moment of failure.

legibility.no_failure_signal legibility score 1 branch B
source: prototype_spec
The participant is asked by the facilitator to verify the populated table against a printed reference sheet provided mid-task. For the recoverable error condition: the agent populates both columns but transposes the 'Pro' and 'Enterprise' tier prices (an error the reference sheet makes detectable). For the irrecoverable error condition: the agent navigates to the correct URL but triggers a destructive action — it overwrites the participant's previously saved table draft with an empty template — and then displays 'Table complete.'

The spec explicitly designs failure visibility into the study through two complementary mechanisms: the printed reference sheet handed to participants mid-task enables detection of the recoverable field-swap error, and the irrecoverable table wipe is visually self-evident as an empty table contradicting the green 'Table complete' message. Additionally, the think-aloud signal and observer coding sheet specifically measure whether and when participants detect the error. The study treats failure visibility as a core observable rather than suppressing it, which is the inverse of the pattern's concern. This is a positive finding: the design actively scaffolds participant detection of system failure.

legibility.no_rationale_at_point_of_action legibility score 1 branch B
source: prototype_spec
The agent displays one of three pre-scripted trust-repair messages determined by the session's condition assignment: (A) Apology — 'I made an error completing that task. I apologize for the mistake and any disruption this caused you.'; (B) Causal Explanation — 'I made an error because the target website restructured its pricing table format after my last training update, causing a field-mapping mismatch. I have logged this for correction.'; (C) Competence Signaling — 'I made an error on this task. Across similar pricing-extraction tasks this session, my accuracy rate has been 94%. I will reattempt with an adjusted extraction strategy.'

Rather than withholding rationale at the point of action, the study's central manipulation is the delivery of post-error rationale in three distinct forms: no rationale (apology), causal rationale (explanation), and competence evidence (signaling). The causal explanation condition provides an explicit mechanistic account of the failure at the moment the participant must decide how to proceed. Additionally, the follow-up response templates (INJECT_FOLLOWUP_RESPONSE_A, B, C) provide further rationale detail on participant request. The design specifically tests whether rationale availability at the point of post-error decision affects reliance choice, which is a direct positive instantiation of the legibility principle this pattern defends.

Provenance legend

Every content-bearing element in the underlying guidebook ends with exactly one of these tags. A markdown-AST linter fails the build if any element is untagged.

[RESEARCHER_INPUT]
The researcher's original premise, verbatim or lightly restated.
[SOURCE_CARD:<id>]
Grounded in a specific hand-curated source card. The id resolves to a file in corpus/source_cards/.
[SIMULATION_REHEARSAL]
Content from the rehearsal. Not evidence. Required to use hedged language.
[AGENT_INFERENCE]
Reasoning by an agent, not grounded in a source or simulation.
[UNCITED_ADJACENT]
Adjacent literature a reviewer named; the researcher must verify before grounding.
[TOOL_VERIFIED]
Measured by a Managed Agents tool-equipped session (bash/grep/file ops).
[IMPORTED_DRAFT]
Researcher's own text lifted verbatim from an imported paper draft (via probe import).
[HUMAN_REQUIRED]
Explicit handoff. Every guidebook must have at least one.
[DO_NOT_CLAIM]
Content the guidebook explicitly flags as unclaimable.
